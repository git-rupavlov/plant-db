# Remote Git Data Architecture

Goal: users run the UI locally, but the plant database lives only in GitHub.

The local app should **not require a cloned repo, SQLite file, or local data copy**. It fetches published data from GitHub at runtime.

Because apparently we are making GitHub cosplay as a database. Honestly, for a small public reference dataset, that is not stupid.

## Core principle

```text
GitHub repository = source of truth
Local UI          = viewer / explorer / visualizer
Local storage     = optional cache only, never authority
```

## Runtime flow

```text
Local browser / desktop UI
        |
        | fetch()
        v
GitHub raw JSON/YAML files
        |
        v
Parse + validate in memory
        |
        v
Render tables, cards, graphs, timelines
```

## Recommended data format

Use YAML for editing by humans, but publish a generated JSON index for UI consumption.

```text
plants/
  cucurbita_moschata.yaml
locations/
  sofia_bg.yaml
public-data/
  index.json
  plants/cucurbita_moschata.json
  locations/sofia_bg.json
```

Why both?

- YAML is nicer for humans editing plant records.
- JSON is easier and safer for frontend fetching.
- The UI should consume JSON, not parse arbitrary YAML in the browser unless needed.

## Read path

The local UI fetches data from raw GitHub URLs:

```text
https://raw.githubusercontent.com/git-rupavlov/plant-db/main/public-data/index.json
https://raw.githubusercontent.com/git-rupavlov/plant-db/main/public-data/plants/cucurbita_moschata.json
```

## Write path

Normal users should not write directly from the UI.

Recommended write workflow:

```text
Edit YAML in GitHub
        |
Pull request / commit
        |
GitHub Action validates data
        |
GitHub Action builds public-data/*.json
        |
Local UI sees new data on next refresh
```

## Local cache policy

The app may cache data for speed/offline preview, but cache must be treated as disposable.

Rules:

- Cache is optional.
- Cache can be cleared at any time.
- Cache never becomes source of truth.
- Cache should store the Git commit SHA or dataset version.

## Versioning

Add a dataset version file:

```json
{
  "dataset": "plant-db",
  "version": "0.1.0",
  "git_ref": "main",
  "generated_at": "manual",
  "species_count": 1,
  "location_count": 1
}
```

Later this can be generated by CI.

## First visualizations

The UI should start with three simple visuals:

1. Phenology timeline
2. Temperature suitability bar
3. Stress impact badges / bars

## Failure modes

The local UI must handle GitHub failures like a civilized tool, not like a JavaScript tragedy.

Handle:

- network unavailable
- GitHub raw file unavailable
- invalid JSON
- schema mismatch
- missing species id
- CORS or rate limit issues

## Practical note

For public repos, raw GitHub access is enough at first.

If the dataset grows or traffic increases, switch to:

- GitHub Pages static JSON
- jsDelivr CDN over GitHub
- Cloudflare Pages
- small API proxy

But not now. Premature infrastructure is how humans summon Kubernetes for a YAML file.
